# 项目开发心得

本笔记纪录一些我写项目的心得 通过自行立项 将好点子转化为好工具 期间遇到的问题 踩得坑 都记录下来 希望对你有所帮助

## 2026-3-25

最近有个好想法就记录下来了 同时付诸实践 啃官方文档做插件去了 看课看不了一点 一点看不了

这个项目时TermRepoPlugin，用来收集文档的

今天在拆分 `extension.ts` 的过程中，深刻体会到：**从“能跑”到“好维护”的关键在于模块化**。

写了一堆功能 堆在extension文件里 导致冗长又复杂 突然就理解了为啥要将项目工程化 分文件 抽离分装成工具

通过将命令、存储、工具函数分离，每个文件只关注一件事，不仅代码可读性飙升，也为后续功能扩展铺平了道路。  

同时，我也发现了当我抽离出某些函数时，必须使用再特定环境下才能收集的信息的参数，例如在extension的context实例中才能获取context.globalStorageUri.fsPath这个路径地址字符串，这是我  **ensureStorageDir.ts检查文件是否存在不存在则创建**  这个函数的重要参数，我也意识到过两天我肯定会忘掉这个函数是干啥的 怎么用 传什么参数 也意识到那些写大项目的大佬应该也是边写代码边写文档的

通过AI工具了解到了JSDoc注释这个东西 也明白了介绍函数使用方法的文档是  API 文档  我要得写一个 为了软件的生态 也为了我自己后续还能看得懂

JSDoc的定义：

> JSDoc 是一种用特定注释格式（`/** ... */`）为 JavaScript/TypeScript 代码编写文档的标准，通过在函数、类等上方添加 `@param`、`@returns` 等标签来描述用途和参数。它能让 IDE 在编码时提供智能提示，还能借助工具（如 TypeDoc）自动生成清晰的 API 文档，从而让代码更易理解、维护和共享。



**边写代码边写 JSDoc 注释**的习惯让“过两天就忘”不再成为困扰——注释既是给未来的自己的说明书，也是团队协作的隐形桥梁。  
而将 `context.globalStorageUri.fsPath` 这类 VS Code 专有依赖通过参数注入的方式抽离，使得工具函数完全独立，可复用、可测试。  
这次重构让我真正理解了“高内聚低耦合” 这个天天吹牛逼的概念是什么 确实有用 慢慢学习吧
